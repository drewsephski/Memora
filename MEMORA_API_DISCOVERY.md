# Memora API Discovery

Discovery date: 2026-06-27

Scope: current repo at `/Users/drewsepeczi/supavec`. This is codebase truth only. I did not inspect or rely on external docs.

## 1. Actual API endpoints

### Official Express API

Source entrypoint: `packages/api/src/server.ts` mounts `packages/api/src/routes/index.ts` at `/`.

Global behavior:

- Middleware: `helmet()`, `cors()`, `morgan("dev")`, `express.json()`, IP rate limit, router, then `errorHandler`.
- IP rate limit: `packages/api/src/middleware/rate-limit.ts` allows 10 requests per 10 seconds per IP, except `GET /` and `GET /healthz`.
- Protected routes use `apiKeyAuth()` then `apiUsageLimit()`.
- API key header: `Authorization: <api-key>` or `Authorization: Bearer <api-key>`; both are accepted by `getApiKeyFromAuthorizationHeader()`.
- API key lookup: `api_keys.api_key = <provided key>`.
- Usage limit lookup: `api_keys(id, team_id, user_id, is_unlimited)` and `profiles` subscription fields. Non-unlimited keys require `user_id`; unlimited keys skip usage checks.
- Error envelopes are inconsistent. Auth middleware returns `{ status: "error", message }`; usage/rate-limit and controllers usually return `{ success: false, error }` or `{ success: false, message }`.

#### `POST /upload_text`

- Source: `packages/api/src/routes/index.ts`
- Middleware: `apiKeyAuth()`, `apiUsageLimit()`, `validateUploadTextRequestMiddleware()`
- Controller: `uploadText` in `packages/api/src/controllers/upload-text.ts`
- Auth: authenticated by Memora API key; requires `api_keys.team_id`.
- Required headers:
  - `Authorization: <MEMORA_API_KEY>` or `Authorization: Bearer <MEMORA_API_KEY>`
  - `Content-Type: application/json`
- Query params: none
- Request body, exactly one of `contents` or `segments`:
  ```ts
  type UploadTextRequest =
    | {
        contents: string; // min length 5
        name?: string; // default "Untitled Document"
        chunk_size?: number; // positive, default 1000
        chunk_overlap?: number; // positive, default 200, must be < chunk_size
      }
    | {
        segments: Array<{
          content: string; // min length 1
          metadata?: Record<string, unknown>;
        }>;
        name?: string; // default "Untitled Document"
      };
  ```
- Behavior:
  - Raw `contents` are chunked by `RecursiveCharacterTextSplitter`.
  - `segments` are treated as caller-provided chunks.
  - Stores source text in Supabase Storage bucket `user-documents` at `/<teamId>/<fileId>.txt` or `/<teamId>/<fileId>.json`.
  - Inserts chunks into `documents` through `storeDocumentsWithFileId()`.
  - Inserts a `files` row with `type: "text"`, `file_id`, `file_name`, `team_id`, and `storage_path`.
- Metadata:
  - Raw text chunks get `{ source: name, file_id }`.
  - Segment metadata is preserved and extended with `{ source: name, file_id }`.
  - No root-level upload metadata field exists.
- Response `200`:
  ```ts
  {
    success: true;
    message: "Text uploaded and processed successfully";
    file_id: string; // UUID
  }
  ```
- Possible errors:
  - `401` missing/invalid API key from auth middleware.
  - `429` IP rate limit or monthly API usage limit.
  - `400` invalid request parameters from Zod validation.
  - `500` storage/vector/file insert failures.
- Classification: authenticated public API.
- Starter stability: starter-ready for raw text and caller-provided segments, with caution that metadata filtering is not exposed by retrieval endpoints.

#### `POST /upload_file`

- Source: `packages/api/src/routes/index.ts`
- Middleware: `apiKeyAuth()`, `apiUsageLimit()`, `upload.single("file")`, `validateUploadRequestMiddleware()`
- Controller: `uploadFile` in `packages/api/src/controllers/upload-file.ts`
- Auth: authenticated by Memora API key; requires `api_keys.team_id`.
- Required headers:
  - `Authorization: <MEMORA_API_KEY>` or `Bearer`
  - `Content-Type: multipart/form-data` generated by the HTTP client
- Query params:
  ```ts
  {
    chunk_size?: number; // coerced positive number, default 1000
    chunk_overlap?: number; // coerced positive number, default 200, must be < chunk_size
  }
  ```
- Request body: multipart form with field `file`.
- Accepted file types:
  - `application/pdf`
  - `text/plain`
  - `text/markdown`
  - any original filename ending in `.md`
- Max file size: 20 MB.
- Behavior:
  - Uploads original file to `user-documents` at `/<teamId>/<fileId>.<txt|md|pdf>`.
  - Text/markdown read as UTF-8; PDFs loaded with LangChain `PDFLoader`.
  - Splits into chunks with `RecursiveCharacterTextSplitter`.
  - Adds `file_id` and `team_id` to chunk metadata.
  - Inserts chunks into `documents`.
  - Inserts a `files` row.
- Response `200`:
  ```ts
  {
    success: true;
    message: string;
    file_name: string;
    file_id: string; // UUID
    chunks: number;
    chunk_size: number;
    chunk_overlap: number;
  }
  ```
- Possible errors:
  - `401` missing/invalid API key.
  - `429` rate/usage limit.
  - `400` invalid query params or missing file.
  - Unsupported file type currently flows through Multer to the global error handler and is likely `500` with `{ status: "error", message: "Only PDF, text, and markdown files are allowed" }`.
  - `500` storage/vector/file insert/PDF processing failures.
- Classification: authenticated public API.
- Starter stability: usable but needs caution. It works, but file ingestion has more operational failure modes and less polished error behavior than text upload.

#### `POST /search`

- Source: `packages/api/src/routes/index.ts`
- Controller: `search` in `packages/api/src/controllers/search.ts`
- Auth: authenticated by Memora API key; requires `api_keys.team_id`.
- Required headers:
  - `Authorization: <MEMORA_API_KEY>` or `Bearer`
  - `Content-Type: application/json`
- Query params: none
- Request body:
  ```ts
  {
    query: string; // min length 1
    file_ids: string[]; // UUIDs, min length 1, de-duplicated
    k?: number; // positive integer, default 3
    include_embeddings?: boolean; // default false
  }
  ```
- Behavior:
  - Validates API key and team.
  - Verifies every requested `file_id` exists in `files` for that team and has `deleted_at is null`.
  - Embeds the query with `createEmbeddings()`.
  - Calls Supabase RPC `match_documents(query_embedding, match_count, filter)` with `filter: { file_id: { in: file_ids } }`.
- Response `200`:
  ```ts
  {
    success: true;
    documents: Array<{
      chunk_id: string;
      file_id: string;
      file_name: string | null;
      content: string;
      score: number; // similarity rounded to 3 decimals
      metadata: Record<string, unknown>;
      embedding?: unknown; // only when include_embeddings true
    }>;
    latency_ms: number;
  }
  ```
- Possible errors:
  - `401` missing/invalid API key.
  - `429` rate/usage limit.
  - `400` invalid body.
  - `403` file ownership mismatch.
  - `500` RPC/embedding/internal errors.
- Classification: authenticated public API.
- Starter stability: starter-ready. This is the best retrieval endpoint for a starter because it returns retrieved context and source metadata.

#### `POST /chat`

- Source: `packages/api/src/routes/index.ts`
- Middleware: `apiKeyAuth()`, `apiUsageLimit()`, `validateChatRequestMiddleware()`
- Controller: `chat` in `packages/api/src/controllers/chat.ts`
- Auth: authenticated by Memora API key; requires `api_keys.team_id` and `api_keys.user_id`.
- Required headers:
  - `Authorization: <MEMORA_API_KEY>` or `Bearer`
  - `Content-Type: application/json`
- Query params: none
- Request body:
  ```ts
  {
    query: string; // min length 1
    file_ids: string[]; // UUIDs, min length 1
    k?: number; // positive integer, default 3 in validator
    stream?: boolean; // default false
  }
  ```
- Behavior:
  - Verifies every requested file belongs to the API key's team.
  - Creates a LangChain `SupabaseVectorStore` filtered by `file_id`.
  - Runs similarity search with `topK = Math.max(k, 8)`, so asking for fewer than 8 still retrieves 8 passages.
  - Generates an answer with OpenRouter using the configured chat model.
- Non-streaming response `200`:
  ```ts
  {
    success: true;
    answer: string;
  }
  ```
- Streaming response `200`:
  - Uses Vercel AI SDK `pipeDataStreamToResponse`.
  - Starts with data events like `{ status: "initialized" }`, streams model output, then writes `{ status: "completed" }`.
  - Not JSON in the same shape as non-streaming.
- Possible errors:
  - `401` missing/invalid API key.
  - `429` rate/usage limit.
  - `400` invalid body.
  - `403` file ownership mismatch.
  - `500` retrieval/model/internal errors.
- Sources/citations:
  - No sources, chunk ids, file ids, or citations are returned from `/chat`.
- Classification: authenticated public API.
- Starter stability: usable but needs caution. It is useful for a beginner chat demo, but not enough for a source-grounded official starter unless the starter also calls `/search` for inspectable citations/context.

## 2. RAG flow

### Text upload

Implementation: `packages/api/src/controllers/upload-text.ts`

1. `POST /upload_text` validates the JSON body in `packages/api/src/middleware/upload-text/validate-request.ts`.
2. The request must include raw `contents` or pre-chunked `segments`, but not both.
3. The API key maps to a Memora `team_id`; this team id determines storage path and file ownership.
4. The controller creates a UUID `fileId`.
5. Raw `contents` are chunked by `RecursiveCharacterTextSplitter` using `chunk_size` and `chunk_overlap`.
6. `segments` are used directly as chunk documents.
7. Source text is uploaded to Supabase Storage bucket `user-documents`.
8. Chunks are embedded and inserted into `documents` by `storeDocumentsWithFileId()`.
9. A `files` row is inserted with `file_id`, `type`, `file_name`, `team_id`, and `storage_path`.
10. Response returns the stable `file_id`.

### File upload

Implementation: `packages/api/src/controllers/upload-file.ts`

1. `POST /upload_file` accepts multipart `file`.
2. Multer memory storage enforces a 20 MB limit and accepts PDF, plain text, and markdown.
3. The file is uploaded to Supabase Storage under `/<teamId>/<fileId>.<ext>`.
4. Text/markdown are read directly; PDFs are loaded with `PDFLoader`.
5. Documents are split with `RecursiveCharacterTextSplitter`.
6. Chunk metadata is extended with `file_id` and `team_id`.
7. Chunks are embedded and inserted into `documents`.
8. A `files` row is inserted.

### Embeddings and storage

Implementation:

- `packages/api/src/utils/vector-store.ts`
- `packages/api/src/utils/embeddings.ts`
- `packages/api/src/utils/ai-config.ts`
- `supabase/migrations/20250217121553_remote_schema.sql`
- `supabase/migrations/20250312121051_file_id_column.sql`
- `supabase/migrations/20250313084417_update_match_documents.sql`

Embedding details:

- Embeddings use `OpenAIEmbeddings` configured against OpenRouter base URL `https://openrouter.ai/api/v1`.
- Default embedding model is `openai/text-embedding-3-small`.
- The `documents.embedding` column is `vector(1536)`.
- Batch insert size is 100 documents.

Database tables:

- `documents`: `id`, `content`, `metadata`, `embedding`, `created_at`, `deleted_at`, plus later-added `file_id`.
- `files`: `id`, `type`, `file_id`, `user_id`, `created_at`, `file_name`, `team_id`, `storage_path`, `deleted_at`.
- `api_keys`: `id`, `api_key`, `created_at`, `user_id`, `team_id`, plus `is_unlimited`.
- `teams`, `team_memberships`, and `profiles` provide the Memora account/team model.

### Retrieval

Implementation:

- `packages/api/src/controllers/search.ts`
- `supabase/migrations/20250313084417_update_match_documents.sql`

Flow:

1. `/search` validates `query`, `file_ids`, `k`, and `include_embeddings`.
2. It verifies all `file_ids` belong to the API key's `team_id` and are not soft-deleted.
3. It embeds the query.
4. It calls `match_documents` with `filter: { file_id: { in: file_ids } }`.
5. `match_documents` filters `deleted_at is null` and, for `file_id.in`, filters the dedicated `documents.file_id` column.
6. Results are ordered by vector distance and returned as chunks.

Metadata filtering:

- The database RPC supports a generic `metadata @> filter` fallback when the filter is not `file_id.in`.
- The public `/search`, `/chat`, and `/embeddings` routes do not expose arbitrary metadata filter parameters.
- Therefore, external apps can attach per-segment metadata at upload time, and it will be returned from `/search`, but they cannot query/filter by arbitrary metadata through the public API today.

### Answer generation

Implementation: `packages/api/src/controllers/chat.ts`

- Memora has a `/chat` endpoint that retrieves context and generates an LLM answer.
- It does not return retrieved sources.
- It uses OpenRouter through the Vercel AI SDK.
- Default chat model is `google/gemini-2.5-flash-lite`.
- For starter apps that need citations, call `/search` and either:
  - display retrieved chunks directly, or
  - call your own LLM with those chunks and keep the chunk metadata as citations.

### Citations and sources

- `/search` returns enough source material for citations: `chunk_id`, `file_id`, `file_name`, `content`, `score`, and `metadata`.
- `/chat` returns only `answer`.
- There is no first-class citation/source array in `/chat`.

## 3. User isolation and multi-tenancy

### What Memora supports

Memora supports:

- `team_id` through API key ownership.
- `file_id` as the document handle used for retrieval.
- `metadata` on chunks, only accepted when using `/upload_text` with `segments`.

Memora does not currently support these as public request-level isolation parameters:

- `user_id`
- `namespace`
- `workspace_id`
- `project_id`
- arbitrary metadata filter

### Where isolation is validated

Validation points:

- `apiKeyAuth()` checks the raw API key exists in `api_keys`.
- `apiUsageLimit()` loads `api_keys.team_id` and `api_keys.user_id`.
- `/search` validates requested `file_ids` are active `files` rows under the API key's `team_id`.
- `/chat` validates requested `file_ids` are under the API key's `team_id`.
- `/embeddings` validates requested `file_ids` are under the API key's `team_id`.
- `/overwrite_text` validates active file ownership under the API key's `team_id`.
- `/delete_file` fetches by `file_id` and `team_id` before deletion.
- `/user_files` lists all active files for the API key's `team_id`.

### What actually enforces isolation

Isolation is enforced by Memora API key -> Memora team -> file ownership.

Important details:

- `files.user_id` exists in the schema, but API uploads do not set it. Upload controllers insert `team_id`, not `user_id`.
- `documents` does not have a first-class `team_id` column. It has `file_id`; some upload paths also place `team_id` in chunk metadata, but raw `/upload_text` does not.
- The service-role Supabase client is used by the API, so Postgres RLS is not the API's main enforcement point. The controllers' explicit team/file checks are the enforcement point.
- The `match_documents` RPC is granted to `anon`, `authenticated`, and `service_role` in the migration. The app should not expose Supabase credentials that let clients call this directly.

### Can a Supabase Auth user id be passed as the Memora user identifier?

No. There is no public Memora API field that accepts a Supabase Auth user id for isolation.

For an official Supabase + Memora starter, treat the Supabase user id as an application-owned identifier. Store it in the starter's own Supabase tables alongside Memora `file_id`s. Do not assume Memora will isolate per Supabase user inside one shared Memora API key.

### What metadata should an external app send?

For MVP:

- Store user/project/document metadata in the starter's Supabase database, not only in Memora.
- For `/upload_text` with `segments`, optional segment metadata can include display/source fields such as:
  ```ts
  {
    source_type: "note" | "document" | "transcript";
    source_id: string; // starter DB id
    title: string;
    user_id: string; // useful for returned context, but not security
  }
  ```
- Treat this metadata as descriptive only. It is returned by `/search`; it is not a security boundary and is not filterable through the public API.

### How an external app should query one user's documents

Recommended flow:

1. Supabase-authenticated user uploads content to the starter.
2. Starter server route/action calls Memora `/upload_text` or `/upload_file` with server-only `MEMORA_API_KEY`.
3. Starter stores returned `file_id` in a Supabase table with `user_id = auth.uid()`.
4. On query, starter server route/action selects only `file_id`s owned by `auth.uid()`.
5. Starter calls Memora `/search` or `/chat` with only those `file_id`s.
6. Browser never sends or sees `MEMORA_API_KEY`.

### Security risks and missing enforcement points

- If a shared Memora API key is used for all starter users and the browser can choose arbitrary `file_ids`, one user can query another user's Memora documents. The starter must enforce `auth.uid() -> file_id` ownership server-side.
- Do not copy dashboard code that passes `apiKey` from browser components into `/api/protected/search` or `/api/protected/chat`.
- `/user_files` lists all files for a Memora team. It is not safe as an end-user document list for a multi-user starter using one shared Memora API key.
- Metadata is not enforced for isolation.
- `/chat` lacks citations, which makes source-grounding hard to audit.
- `/delete_file`, `/overwrite_text`, and `/resync_file` perform multi-step updates/deletes and should be guarded by starter-owned file ownership checks.

## 4. Client integration contract for a Next.js + Supabase app

### Required environment variables

Server-only:

```bash
MEMORA_API_URL=https://memora-api-drew.fly.dev
MEMORA_API_KEY=...
```

Supabase starter variables:

```bash
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
SUPABASE_SERVICE_ROLE_KEY=... # only if the starter has admin/server maintenance tasks
```

Never expose `MEMORA_API_KEY` to the browser. Avoid `NEXT_PUBLIC_MEMORA_API_KEY`.

### Base API URL

Current code defaults to:

```txt
https://memora-api-drew.fly.dev
```

Use `MEMORA_API_URL` in the starter so it can move to the eventual official hostname without code changes.

### Auth header format

Use:

```ts
Authorization: process.env.MEMORA_API_KEY
```

Bearer also works:

```ts
Authorization: `Bearer ${process.env.MEMORA_API_KEY}`
```

The existing repo generally uses the raw key.

### Minimum viable server-side client

Recommended functions:

```ts
type MemoraUploadTextRequest =
  | {
      name?: string;
      contents: string;
      chunk_size?: number;
      chunk_overlap?: number;
    }
  | {
      name?: string;
      segments: Array<{
        content: string;
        metadata?: Record<string, unknown>;
      }>;
    };

type MemoraUploadTextResponse = {
  success: true;
  message: string;
  file_id: string;
};

type MemoraSearchRequest = {
  query: string;
  file_ids: string[];
  k?: number;
  include_embeddings?: boolean;
};

type MemoraSearchDocument = {
  chunk_id: string;
  file_id: string;
  file_name: string | null;
  content: string;
  score: number;
  metadata: Record<string, unknown>;
  embedding?: unknown;
};

type MemoraSearchResponse = {
  success: true;
  documents: MemoraSearchDocument[];
  latency_ms: number;
};

type MemoraChatRequest = {
  query: string;
  file_ids: string[];
  k?: number;
  stream?: false;
};

type MemoraChatResponse = {
  success: true;
  answer: string;
};

type MemoraFile = {
  type: string | null;
  file_id: string | null;
  created_at: string;
  file_name: string | null;
  team_id: string | null;
};
```

Expose these functions from a server-only module:

- `uploadText(input: MemoraUploadTextRequest): Promise<MemoraUploadTextResponse>`
- `search(input: MemoraSearchRequest): Promise<MemoraSearchResponse>`
- `chat(input: MemoraChatRequest): Promise<MemoraChatResponse>`
- `deleteFile(fileId: string): Promise<void>`
- Optional/admin only: `uploadFile(formData)`, `listTeamFiles()`, `overwriteText()`

### Example fetch calls

```ts
const memoraHeaders = {
  Authorization: process.env.MEMORA_API_KEY!,
  "Content-Type": "application/json",
};

export async function uploadTextToMemora(contents: string, name: string) {
  const response = await fetch(`${process.env.MEMORA_API_URL}/upload_text`, {
    method: "POST",
    headers: memoraHeaders,
    body: JSON.stringify({ contents, name }),
  });

  const payload = await response.json();
  if (!response.ok || payload.success !== true) {
    throw new Error(payload.error || payload.message || "Memora upload failed");
  }

  return payload as MemoraUploadTextResponse;
}

export async function searchMemora(query: string, fileIds: string[]) {
  const response = await fetch(`${process.env.MEMORA_API_URL}/search`, {
    method: "POST",
    headers: memoraHeaders,
    body: JSON.stringify({ query, file_ids: fileIds, k: 5 }),
  });

  const payload = await response.json();
  if (!response.ok || payload.success !== true) {
    throw new Error(payload.error || payload.message || "Memora search failed");
  }

  return payload as MemoraSearchResponse;
}
```

### Recommended error handling

- Parse JSON when possible; fall back to response text.
- Prefer a normalized local error type:
  ```ts
  type MemoraClientError = {
    status: number;
    message: string;
    details?: unknown;
  };
  ```
- Map:
  - `401`: server misconfiguration or invalid Memora key.
  - `403`: file ownership mismatch; likely starter bug or attempted cross-user access.
  - `429`: rate limit or plan limit; show retry/upgrade messaging.
  - `400`: validation error; show user-correctable input problem.
  - `500`: Memora/internal failure; log server-side and show generic user message.

### What should never be exposed to the browser

- `MEMORA_API_KEY`
- `SUPABASE_SERVICE_ROLE_KEY`
- Raw list of all Memora team files when the starter has multiple end users
- Arbitrary Memora `file_id`s not owned by the current Supabase user

## 5. Endpoint quality assessment

| Capability | Endpoint/status | Assessment | Reason |
|---|---|---|---|
| Health | `GET /`, `GET /healthz` | starter-ready | Simple public health checks. |
| Upload text | `POST /upload_text` | starter-ready | Cleanest ingestion path; supports raw text and pre-chunked segments. |
| Upload file | `POST /upload_file` | usable but needs caution | Supports PDF/text/markdown, but multipart/PDF errors and file processing add complexity. |
| Search/query | `POST /search` | starter-ready | Best official retrieval contract; returns chunks, scores, metadata, and optional embeddings. |
| Embeddings retrieval | `POST /embeddings` | not recommended | Compatibility/demo endpoint with weaker response and unused flags. Prefer `/search`. |
| Chat/answer generation | `POST /chat` | usable but needs caution | Generates answers but no citations/sources; streaming shape differs from JSON shape. |
| List files | `POST /user_files` | usable but needs caution | Team-level listing, not safe as per-user listing in a shared-key starter. Count includes deleted rows. |
| Delete document/file | `POST /delete_file` | usable but needs caution | Works after team ownership check, but destructive and error statuses are imperfect. |
| Resync/reprocess | `POST /resync_file` | internal only | Operational maintenance; markdown handling appears incorrect. Not MVP starter material. |
| Overwrite/update text | `POST /overwrite_text` | usable but needs caution | Text-only; multi-step update; include after MVP if editing is needed. |
| Metadata filtering | missing public endpoint support | missing / should be added | RPC can filter metadata, but public APIs only accept `file_ids`. |
| Citation/source output | `/search` only | partial | `/search` returns source chunks; `/chat` returns no source array. |
| Cache stats | no route found | missing / should be added or docs removed | AGENTS/CLAUDE mention `GET /cache_stats`, but `routes/index.ts` does not register it. |
| `/embed` | no route found | not recommended/docs outdated | Public content mentions `/embed`; implementation uses `/upload_text` or `/upload_file`. |
| `/query` | no route found | not recommended/docs outdated | Public content mentions `/query`; implementation uses `/search`. |

## 7. Recommended official starter scope

### Definitely implement

- Supabase Auth sign-in.
- A starter-owned `memora_documents` table:
  ```sql
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  memora_file_id uuid not null,
  title text not null,
  source_type text not null default 'text',
  created_at timestamptz not null default now()
  ```
- RLS on `memora_documents` so users can only read/delete their own rows.
- Server action or route to upload text:
  - authenticate Supabase user
  - call Memora `/upload_text`
  - store returned `file_id` with `auth.uid()`
- Server route/action to search:
  - authenticate Supabase user
  - select only that user's `memora_file_id`s
  - call Memora `/search`
  - return normalized chunks to browser
- Optional server route/action to chat:
  - either call `/chat` for a simple answer
  - or call `/search` and then the starter's LLM for source-aware answers
- Server-side delete:
  - verify row belongs to `auth.uid()`
  - call Memora `/delete_file`
  - delete or soft-delete local mapping

### Avoid for MVP

- `/embeddings`
- `/resync_file`
- `/overwrite_text`
- `/user_files` for end-user document lists
- Browser-provided Memora API keys
- Metadata-based security
- Promising `/chat` citations from Memora
- Cache stats UI until an actual endpoint exists

### UI pages

Minimal starter pages:

1. Sign in / sign up.
2. Dashboard with:
   - text upload form
   - user's uploaded documents from starter Supabase table
   - search box
   - retrieval results with chunk content, score, file title, metadata
3. Optional chat panel:
   - clear label that answers are generated from selected documents
   - show retrieved chunks underneath as sources if using `/search`

### File upload or text upload?

Start with text upload.

Reason: `/upload_text` is more deterministic, supports metadata via segments, and avoids PDF/multipart handling. Add file upload as a second example once the core isolation pattern is solid.

### Chat, query/search, or both?

Implement search first. Add chat as optional.

Reason: `/search` exposes source chunks and is auditable. `/chat` is convenient but has no citations. A beginner-friendly official starter should teach the safe retrieval contract before answer generation.

### File management?

Include starter-owned listing and delete. Do not call `/user_files` for user-facing lists in a shared-key app.

### Store Memora metadata in Supabase?

Yes. Store at least:

- `memora_file_id`
- Supabase `user_id`
- title/name
- source type
- created timestamp
- optional local status/error fields

This local table is the starter's security boundary.

### Cleanest beginner flow

1. User signs in with Supabase.
2. User pastes text and title.
3. Server action calls Memora `/upload_text`.
4. Server action stores `file_id` in Supabase under `auth.uid()`.
5. User asks a question.
6. Server action loads that user's file IDs and calls `/search`.
7. UI shows retrieved chunks and scores.
8. Optional: server action calls an LLM with those chunks and displays an answer plus sources.

## 8. Source of truth summary

### Confirmed endpoints to use

- `POST /upload_text`
- `POST /search`
- `POST /chat` only for simple no-citation answer generation
- `POST /delete_file` only after starter-owned user/file authorization
- `GET /healthz`

### Confirmed endpoints to avoid or defer

- Avoid `/embeddings` for new starter code; use `/search`.
- Avoid `/user_files` for per-user lists in a shared-key Supabase starter.
- Defer `/upload_file` until after text upload/search is working.
- Defer `/overwrite_text` and `/resync_file`.
- Do not use `/embed`, `/query`, or `/cache_stats`; they are not implemented routes in this repo.

### Recommended server-side client functions

- `uploadText()`
- `search()`
- `chat()` optional
- `deleteFile()` optional
- `uploadFile()` later

### Recommended Supabase integration pattern

Supabase Auth owns end-user identity. The starter owns user-to-Memora-file mapping in Supabase. Memora owns ingestion, embeddings, retrieval, and optional answer generation. The browser talks only to Next.js server routes/actions; Next talks to Memora with `MEMORA_API_KEY`.

### Blockers before an official template

- Decide whether official starter should use `/chat`; if yes, Memora should add source/citation output or the starter should generate answers itself from `/search`.
- Add public metadata filtering if starters are expected to support project/workspace/namespace filtering inside one Memora team.
- Fix docs/copy mentioning `/embed`, `/query`, `GET /search`, `DELETE /delete_file`, and `/cache_stats`.
- Normalize error envelopes and status codes, especially file type validation and delete not-found behavior.
- Consider adding first-class `user_id`/`namespace` support or make the official docs explicit that Memora isolation is team/API-key plus `file_id`, not external-user based.
- Consider making `/user_files` count exclude soft-deleted rows.
